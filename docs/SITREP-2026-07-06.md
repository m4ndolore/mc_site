## What shipped (commits `92fdb94..1b15269` on main)

**The page** (`curriculum.html`): guided mission path styled as Act II of `/start` on the founder-path design system — segmented progress bar, one stage in focus, sticky 5-stage rail, resource cards with time estimates, site navbar/footer, light-mode support.

**The behavior** (`js/curriculum.js`):
- Open a resource = engaged (auto-check); check button independently toggleable (keyboard-accessible)
- Advance unlocks at 2+ engaged resources; any stage peekable via the rail
- Triage mapping: saved `/start` answers (`mc-founder-path-state-v1`) pick the starting stage, with a dismissible banner; `visionary/curious→Preflight`, `operator/builder→Spot`, `team-with-prototype→Ready`, `scaling→Tension`

**The architecture** (why this one works where the SPA didn't):
- One root-level Vite entry like every other page — no sub-build, no base paths, no `_routes.json`
- `scripts/inject-curriculum.mjs` (build step `curriculum:inject`) renders `data/curriculum.json` into the page between `CURRICULUM:*` markers at build time — in place, idempotent, hardened (HTML escaping, `<` JSON-LD escaping, duplicate-id assertion)
- No runtime fetch, no loading state, no API — a blank screen is structurally impossible

## Why v1 failed (for the record)

The React SPA fought the site's architecture at every layer: mc_site is a vanilla multi-page Vite build where every page is an explicit `rollupOptions.input` entry. The SPA needed its own build, base path, asset paths, and routing config, and each mismatch produced a different failure (404 assets, black screens, invisible text, missing data). Its frontend also called `/api/curriculum/*` endpoints that were never built. **Rule going forward: new pages are root-level HTML entries; no SPA sub-builds.**
